Etapa 1 
 

ETAPA 1: DEFINICIÓN DEL PROYECTO Y 
CASO DE NEGOCIO 
1.1 Presentación y Justificación del Proyecto 
El proyecto consiste en el diseño y desarrollo de ParkScan, un sistema automatizado de 
gestión integral para playas de estacionamiento. El núcleo del sistema está diseñado para 
transformar la gestión operativa mediante cuatro pilares fundamentales: control de flujo de 
vehículos, optimización de activos a través del monitoreo en tiempo real, gestión 
transaccional segura y generación de inteligencia de negocio mediante reportes detallados. 
Justificación Técnica y de Negocio: La propuesta surge como respuesta a la necesidad de 
modernizar estacionamientos de distintas dimensiones que actualmente operan de forma 
manual. El conteo manual suele generar errores críticos, demoras operativas y falta de 
transparencia financiera. ParkScan se diferencia por su enfoque de Producto Mínimo Viable 
(MVP): ofrece una alternativa económica y escalable que reduce costos de infraestructura al 
no depender de sensores físicos complejos, utilizando en su lugar un modelo lógico basado 
en tecnología de códigos QR para el registro de estadías. 
 
 
 
 
 
 
 
 
 
 
 

1.2 Análisis del Entorno 
Entorno Interno (La Organización) 
La organización objetivo es una empresa de gestión de estacionamientos con una 
estructura operativa pequeña y un grado de digitalización bajo, donde predominan los 
registros en papel o herramientas informáticas aisladas. A pesar de esto, existe una visión 
estratégica hacia la modernización y la optimización de procesos para reducir el error 
humano. El equipo de desarrollo cuenta con los conocimientos técnicos en programación 
(Python/Flask) y bases de datos (MySQL) necesarios para ejecutar el proyecto, apoyándose 
en software libre para garantizar la viabilidad. 
Entorno Externo (El Mercado) 
El mercado de estacionamientos urbanos en zonas céntricas (como Mendoza o San Rafael) 
es altamente competitivo y está regulado por normativas municipales sobre tarifas y 
seguridad. Si bien existen sistemas de alta gama con barreras automatizadas y sensores de 
masa, su elevado costo de implementación los vuelve inaccesibles para comercios 
medianos. ParkScan ocupa este nicho, proporcionando una herramienta profesional que 
proyecta una imagen corporativa adecuada y cumple con los requisitos de licenciamiento 
abierto. 
 
 
 
 
 
 
 
 
 
 
 

1.3 Objetivos y Alcances Preliminares 
Objetivos Estratégicos 
● Control de Ocupación: Gestionar en tiempo real la disponibilidad de espacios por 
sectores y de forma global para optimizar el uso del predio. 
● Automatización Transaccional: Implementar la emisión y validación de tickets QR 
para agilizar el ingreso, el cálculo exacto de tarifas y el proceso de cobro. 
● Trazabilidad y Auditoría: Generar una base de datos confiable que permita auditar 
la recaudación diaria, segregada por turnos y operarios responsables. 
● Experiencia del Usuario: Ofrecer una interfaz minimalista y funcional que 
simplifique la interacción tanto para el personal de playa como para los conductores. 
Alcance del Software 
El sistema ParkScan integrará las siguientes capacidades funcionales: 
● Arquitectura de Tres Capas (Three-Tier Architecture): El sistema se estructurará 
mediante una separación modular estricta para garantizar la escalabilidad y el 
mantenimiento. 
● Capa de Presentación (Frontend): Interfaces de usuario diferenciadas para 
Administradores, Operarios y Clientes. 
● Capa de Lógica de Negocio (Backend): Desarrollada en Flask (Python), 
encargada de orquestar la lógica transaccional, el cálculo de tarifas y la 
comunicación con servicios externos. 
● Capa de Persistencia (Datos): Una base de datos relacional en MySQL que 
asegura la integridad de los registros operativos y financieros. 
Gestión de Inventario y Activos: Módulos específicos para la administración granular 
de la topología física del estacionamiento. 
● Sectores: Segmentación del predio para una mejor organización logística. 
● Plazas: Identificación individual y monitoreo en tiempo real del estado de cada 
espacio (Libre/Ocupado). 
Módulo de Gestión Transaccional y Pasarela de Pagos (API): Este componente es el 
núcleo financiero del sistema y cumple con el requisito de integración externa. 
 

● Integración de API de Pagos: Consumo de servicios externos (como Mercado 
Pago u otras pasarelas electrónicas) para administrar cobros de forma segura y 
automatizada. 
● Cálculo Dinámico de Tarifas: Algoritmo que procesa el tiempo exacto de estadía 
para generar el monto a abonar, eliminando errores de cálculo manual. 
● Validación de Salida: Sistema que confirma la conciliación automática del pago 
para habilitar la liberación de la barrera de salida. 
Panel Administrativo e Inteligencia de Negocio: Interfaz estratégica para la toma de 
decisiones basada en datos reales. 
● Métricas de Desempeño: Visualización en tiempo real de la ocupación, flujo de 
ingresos y volumen de tráfico. 
● Reportes Operativos: Generación de informes exportables en formatos 
normalizados como PDF y CSV. 
● Auditoría de Cierres Contables: Mecanismo de consolidación financiera diaria que 
agrupa transacciones por turno y operario para garantizar la transparencia total. 
 
 
 
 
 
 
 
 

Etapa 2 
 

ETAPA 2: ANÁLISIS DE FACTIBILIDAD Y 
PROPUESTA TÉCNICA  
2.1 Matriz de Factibilidad Multidimensional 
Para asegurar que el proyecto sea ejecutable y sostenible, se ha realizado un desglose 
exhaustivo de los factores de viabilidad: 
● Factibilidad Técnica: El proyecto es técnicamente posible ya que se apoya en 
tecnologías conocidas y accesibles para el equipo, como Python y Flask para la 
lógica de backend, y MySQL para la persistencia de datos. El equipo cuenta con los 
conocimientos necesarios adquiridos en el cursado para manejar la concurrencia de 
usuarios y la conexión a bases de datos. Además, al no requerir integración con 
software externo complejo inicialmente, se reducen los riesgos técnicos 
significativamente. Los riesgos identificados, como el manejo de concurrencia y la 
integridad de los datos, son manejables mediante la aplicación de buenas prácticas 
de desarrollo y pruebas de estrés. 
● Factibilidad Económica: El proyecto es económicamente viable, ya que no 
requiere inversión en hardware especializado. Todo el desarrollo se realizará con 
software libre o gratuito, como Python, Flask, MySQL y Figma. La inversión principal 
reside en el tiempo de trabajo del equipo. A largo plazo, el costo-beneficio es 
positivo, ya que la automatización optimiza la gestión, evita errores humanos y 
reduce costos operativos en estacionamientos. 
● Factibilidad Operativa: El proyecto puede llevarse a cabo sin inconvenientes 
operativos, dado que el equipo posee competencias en programación, diseño de 
bases de datos y gestión ágil. El sistema no modifica flujos de trabajo complejos; 
simplifica las tareas de control reemplazando el conteo manual por una interfaz 
automatizada. La adopción por parte del personal será fluida debido a una interfaz 
diseñada bajo principios de simplicidad. 
● Factibilidad Legal: El sistema cumple con la normativa legal vigente, ya que no 
recopila información sensible de los usuarios finales. Los datos manejados (patente, 
hora de entrada/salida, sector) son de carácter operativo. Asimismo, se utilizan 
librerías con licencias abiertas como ZXing y PyQRCode, cumpliendo con los 
requisitos de licenciamiento de software libre. 
 

● Factibilidad Política: Actualmente, el proyecto no cuenta con el apoyo de 
Stakeholders externos (interesados reales), por lo que se define como una iniciativa 
interna de carácter académico. 
2.2 Metodología de Desarrollo y Ciclo de Vida del 
Software 
Se ha seleccionado el marco de trabajo Scrum, alineado con un enfoque ágil que permite la 
entrega de valor incremental: 
● Desarrollo Iterativo: El trabajo se divide en sprints cortos para entregar avances 
funcionales constantes, permitiendo una mejor adaptación a cambios y 
retroalimentación continua. 
● Marco PMBOK®: La gestión integral se rige bajo los lineamientos del PMBOK® 
7ma Edición, asegurando que el sistema evolucione de forma consistente hasta su 
despliegue final. 
● Justificación Académica: El enfoque ágil se ajusta de manera óptima a la dinámica 
de trabajo y los tiempos del cuatrimestre universitario. 
2.3 Propuesta Técnica Detallada (Stack Tecnológico) 
La infraestructura tecnológica se ha definido priorizando la robustez, el mantenimiento y la 
escalabilidad: 
● Backend y Lógica: Empleo de Python con el framework Flask, priorizando la 
robustez y la facilidad de mantenimiento. 
● Persistencia de Datos: Uso de MySQL normalizado para asegurar la persistencia, 
administración de transacciones y auditoría. 
● Tecnología QR: Integración de bibliotecas ZXing y PyQRCode para la generación y 
escaneo de códigos bidimensionales. 
● Reportería: Generación de informes operacionales en formatos normalizados como 
PDF y CSV. 
● Integraciones (API): Se prevé la integración de pasarelas de pago electrónico para 
administrar cobros, agilizar el flujo de salida y conciliar automáticamente el estado 
del ticket. 
2.4 Diseño de Interfaz y Arquitectura de Información 
 

El diseño se fundamenta en el minimalismo funcional, dividiendo la arquitectura en cuatro 
componentes esenciales: 
1. Emisión: Generación rápida de tickets al ingreso del vehículo. 
2. Administración: Visualización en tiempo real de la ocupación por sectores. 
3. Caja: Escaneo, cálculo de tarifas y procesamiento de pagos. 
4. Validación: Control final de barrera para garantizar el egreso seguro. 
2.5 Prototipo UI/UX (Mockups) 
El prototipo visual refleja la simplicidad operativa necesaria para el flujo constante de una 
playa de estacionamiento: 
● Interfaz de Entrada: Pantalla de bienvenida con opción clara de "ENTRADA". 
● Ticket Digital: Comprobante que incluye el ID del ticket, la hora de entrada exacta y 
la recomendación de conservarlo para la salida. 
● Interfaz de Salida: Pantalla de despedida ("¡Hasta Pronto!") que confirma la 
finalización del servicio tras el pago.

El sistema establece vínculos lógicos que garantizan que cada operación sea 
rastreable hasta su origen: 
● Infraestructura: Un Sector contiene múltiples Plazas (1:N). Cada Plaza puede 
estar asociada a múltiples Tickets a lo largo del tiempo (1:N). 
 
● Operación: Un Vehículo puede generar múltiples Tickets en distintas visitas 
(1:N). 
 
● Responsabilidad: Un Operario y un Turno específico pueden estar vinculados 
a múltiples Tickets y Pagos (1:N), permitiendo auditar la recaudación 
individual. 
 
 

● Finalización: Existe una relación biunívoca entre Ticket y Pago (1:1), 
asegurando que cada servicio tenga un único comprobante de cancelación 
de deuda. 
Registro contable del sistema 
El sistema incorporará un módulo contable que permitirá registrar cada operación 
realizada dentro del estacionamiento. 
Cada ingreso de vehículo generará un ticket asociado a una plaza, un operario y un 
turno de trabajo. Al momento del egreso, se registrará el pago correspondiente al 
servicio. 
Esta información permitirá: 
● Registrar todos los vehículos ingresados durante el día. 
 
● Registrar los pagos realizados por los clientes. 
 
● Asociar cada pago con el operario que realizó la operación. 
 
● Agrupar los ingresos por turnos de trabajo. 
 
● Calcular la recaudación total diaria del estacionamiento. 
 
 
 
 
Registro y Cierre Contable Diario 
El sistema integra un mecanismo de consolidación financiera que se ejecuta al 
finalizar cada jornada operativa . Este proceso automatizado agrupa todas las 
transacciones monetarias registradas para garantizar la transparencia y la rendición 
de cuentas.
